# @bagaking/jsoneditor

[![CI](https://github.com/bagaking/jsoneditor/actions/workflows/ci.yml/badge.svg)](https://github.com/bagaking/jsoneditor/actions/workflows/ci.yml)

A React JSON editor component for product consoles, rule editors, debug panels, low-code forms, and any place where JSON needs to stay editable without losing validation, schema context, or UI affordances. It packages CodeMirror 6 editing, AJV JSON Schema validation, path decoration, toolbar/status UI, and TypeScript types behind one component library entry.

📚 [Full documentation](https://bagaking.github.io/jsoneditor) | [中文](./README.md)

## Component Contract

- **React contract**: `react` and `react-dom` are peer dependencies and support `^18.3.0 || ^19.0.0`.
- **Editor contract**: CodeMirror 6, AJV, and `ajv-formats` are package dependencies; the Vite library build keeps them as runtime imports for the consuming bundler to resolve.
- **Style contract**: import `@bagaking/jsoneditor/style.css` from your app entry when you need the built-in component styles. The published `dist/style.css` only covers built-in styles; consumer Tailwind classes passed through `decorationConfig` must be generated by your own Tailwind content scan or safelist.
- **Package contract**: the package exports ESM, CommonJS, TypeScript declarations, and `./style.css`. The root entry exports `JsonEditor`, `EditorCore`, public config types, and built-in action icon helpers.

## Installation

```bash
pnpm add @bagaking/jsoneditor
# or
npm install @bagaking/jsoneditor
# or
yarn add @bagaking/jsoneditor
```

```tsx
import '@bagaking/jsoneditor/style.css';
import { JsonEditor } from '@bagaking/jsoneditor';
```

`dist/style.css` does not generate CSS for your app-specific Tailwind classes. If path decorations use classes such as `bg-blue-100/30`, `rounded`, or `text-green-600`, include the relevant source files or a safelist in your app's Tailwind configuration.

## Quick Start

`JsonEditor` initializes content with `defaultValue` and reports string changes through `onValueChange`. Use a `ref` to access `EditorCore` when external controls need to read or write editor state.

```tsx
import { useState } from 'react';
import '@bagaking/jsoneditor/style.css';
import { JsonEditor } from '@bagaking/jsoneditor';

export function ConfigEditor() {
  const [value, setValue] = useState('{\n  "env": "prod"\n}');

  return (
    <JsonEditor
      defaultValue={value}
      onValueChange={setValue}
      onError={(error) => console.error(error)}
      codeSettings={{
        fontSize: 14,
        lineNumbers: true,
        bracketMatching: true,
        autoCompletion: true
      }}
      themeConfig={{ theme: 'light' }}
      toolbarConfig={{
        features: {
          format: true,
          minify: true,
          validate: true,
          copy: true
        }
      }}
    />
  );
}
```

## API Use Cases

### Schema Validation

Pass a JSON Schema to validate JSON with AJV inside the editor. Errors and field context are exposed through `onError`, the status bar, and the schema information panel.

```tsx
import { JsonEditor } from '@bagaking/jsoneditor';

const schema = {
  type: 'object',
  properties: {
    name: {
      type: 'string',
      minLength: 1,
      description: 'Project name'
    },
    version: {
      type: 'string',
      pattern: '^\\d+\\.\\d+\\.\\d+$',
      description: 'Semver version'
    }
  },
  required: ['name', 'version']
};

export function PackageJsonEditor() {
  return (
    <JsonEditor
      defaultValue={JSON.stringify({ name: 'demo', version: '1.0.0' }, null, 2)}
      schemaConfig={{ schema }}
      validationConfig={{ validateOnChange: true }}
    />
  );
}
```

See the [Schema Validation Guide](./docs/api/schema-validation.md) for more detail.

### Path Decoration

`decorationConfig.paths` can style keys, values, or both for a JSON path, and can attach icons or click handlers. Built-in style names such as `underline`, `bold`, and `italic` are handled by the component; application Tailwind classes are built by the consumer.

```tsx
import { JsonEditor, rocketActionIcon } from '@bagaking/jsoneditor';

export function DecoratedEditor() {
  return (
    <JsonEditor
      defaultValue={JSON.stringify({ version: '1.0.0', status: 'active' }, null, 2)}
      decorationConfig={{
        paths: {
          '$["version"]': {
            target: 'value',
            style: 'italic bg-blue-100/30 rounded px-1',
            onClick: (value) => console.log('version', value)
          },
          '$["status"]': {
            target: 'both',
            style: 'text-green-600 font-medium',
            icon: rocketActionIcon
          }
        }
      }}
    />
  );
}
```

See the [Decoration System Documentation](https://bagaking.github.io/jsoneditor/api/decoration) for more detail.

### Imperative Ref

`EditorCore` exposes methods such as `getValue`, `setValue`, `getCursorPosition`, `getSchemaPathAtPosition`, `getSchemaAtPath`, `getValueAtPath`, `setValueAtPath`, `getLineEndPosition`, `addExtension`, `removeExtension`, `scrollToLine`, `updateConfig`, and `destroy`. This is useful for external buttons, form submission, or debugging tools.

```tsx
import { useRef } from 'react';
import { JsonEditor, type EditorCore } from '@bagaking/jsoneditor';

export function RefDrivenEditor() {
  const editorRef = useRef<EditorCore>(null);

  return (
    <>
      <button
        type="button"
        onClick={() => console.log(editorRef.current?.getValue())}
      >
        Log JSON
      </button>
      <button
        type="button"
        onClick={() => editorRef.current?.setValueAtPath('$["hello"]', 'reader')}
      >
        Update hello
      </button>
      <JsonEditor
        ref={editorRef}
        defaultValue={'{"hello":"world"}'}
      />
    </>
  );
}
```

### CodeMirror Extensions

Pass extra CodeMirror behavior through the `extensions` prop. The component forwards those extensions to the underlying `EditorCore`.

```tsx
import type { Extension } from '@codemirror/state';
import { JsonEditor } from '@bagaking/jsoneditor';

export function ExtendedEditor({ extensions }: { extensions: Extension[] }) {
  return (
    <JsonEditor
      defaultValue="{}"
      extensions={extensions}
    />
  );
}
```

## Release Readiness

Maintainers should run:

```bash
pnpm install --frozen-lockfile
pnpm verify
```

`pnpm verify` runs the repository hygiene check, library build, public entrypoint smoke check, demo build, and package dry run. CI uses the same entrypoint so local release readiness and GitHub Actions do not drift.

`pnpm check:repo` rejects accidentally tracked build output, OS metadata, local home-directory absolute paths, and obvious credential-like assignments. `pnpm build` cleans `dist`, runs TypeScript compilation, runs the Vite library build, and builds Tailwind output for `dist/style.css`. The expected outputs include `dist/index.js`, `dist/index.cjs`, TypeScript declarations, and `dist/style.css`.

`pnpm pack:dry-run` runs `scripts/check-pack-manifest.mjs`. The script first requires the expected `dist` artifacts, then invokes `npm pack --dry-run --json` and checks that the package contains the required entry files, declarations, style file, README, and LICENSE. It also rejects unexpected `dist` artifacts, unreachable CommonJS chunks, bundled `react-dom/client` source, and ESM/CJS outputs that fail to keep `react-dom/client` as an external import/require.

`pnpm check:entrypoints` runs `scripts/check-public-entrypoints.mjs`. It installs the current workspace package into a temporary consumer project and verifies ESM import, CommonJS require, the stylesheet subpath, and TypeScript declaration consumption.

This gate validates the package manifest and build artifact boundary. It does not replace unit tests, browser interaction tests, docs-site checks, or proof that a consuming app generated its own Tailwind classes.
